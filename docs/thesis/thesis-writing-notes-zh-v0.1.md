## 3. 为什么采用当前论文结构

我最终采用了典型的软件工程毕业论文结构，而不是单纯按算法论文结构撰写，主要原因如下：

1. 当前项目本质上是“评测框架与工程系统实现”，不是单一模型训练论文
2. 软件工程本科论文更强调需求分析、总体设计、关键实现、测试与应用分析
3. 参考的 Word 示例论文本身也采用了“绪论—需求分析与架构—关键实现—测试评估—案例分析—总结展望”的工程型结构
4. 当前项目包含较强的系统工程内容，例如统一适配器、运行脚本、可观测性、工件复用和结果落盘，这些内容非常适合按工程论文结构展开

因此，论文被组织为以下八章：

1. 绪论
2. 需求分析与总体设计
3. 评测任务与指标体系设计
4. 三探针归因评测框架设计
5. 多系统适配与工程实现
6. 实验设计与结果分析
7. 案例分析与工程讨论
8. 总结与展望

## 4. 各章节与项目实现的对应关系

### 4.1 第一章 绪论

这一章主要对应：

1. 参考文档中的研究背景材料
2. 长期记忆评测相关论文阅读材料
3. 项目在 README 和架构文档中对整体目标的描述

本章作用是交代为什么“长期记忆系统评测”值得做，以及为什么当前项目要从单纯正确率评测升级到三探针归因评测。

### 4.2 第二章 需求分析与总体设计

这一章主要对应：

1. `README.md` 中对项目分层的说明
2. `docs/最终指标.md` 中对评测目标的定义
3. 项目中 dataset、adapters、eval_core、pipeline 的真实分层实现

本章的写法刻意偏软件工程风格，强调功能需求、非功能需求、可扩展性、可解释性与可复现性。

### 4.3 第三章 评测任务与指标体系设计

这一章主要对应：

1. `docs/最终指标.md`
2. `docs/architecture/eval-framework-next-step-solution-zh-v0.1.md`
3. `src/memory_eval/eval_core/models.py`
4. `src/memory_eval/eval_core/correctness_judge.py`

本章的重点是把项目中的 POS/NEG 任务、状态空间、缺陷编码和 `final_correct` 统计方式整理为论文语言，而不是直接照搬代码说明。

### 4.4 第四章 三探针归因评测框架设计

这一章主要对应：

1. `src/memory_eval/eval_core/encoding_agent.py`
2. `src/memory_eval/eval_core/retrieval_agent.py`
3. `src/memory_eval/eval_core/generation_agent.py`
4. `src/memory_eval/eval_core/attribution_agent.py`
5. `src/memory_eval/eval_core/engine.py`

本章是论文方法部分的核心，强调：

1. 为什么要拆成编码、检索、生成三层
2. 为什么最终还需要归因代理收敛结果
3. 为什么不能只看最终答案是否正确

### 4.5 第五章 多系统适配与工程实现

这一章主要对应：

1. `src/memory_eval/adapters/base.py`
2. `src/memory_eval/adapters/o_mem_adapter.py`
3. `src/memory_eval/adapters/membox_adapter.py`
4. `src/memory_eval/adapters/registry.py`
5. `src/memory_eval/pipeline/runner.py`
6. `scripts/run_real_memory_eval.py`

这一章突出“软件工程实现价值”，因为项目真正有难点的地方不仅在方法设计，更在于如何把不同记忆系统纳入一个统一评测协议。

### 4.6 第六章 实验设计与结果分析

这一章目前是“结构完整、结果留空”的状态。这是根据你的要求专门处理的，因为当前定量结果尚未全部跑出，不能伪造实验数值。

本章已经写入：

1. 实验目标
2. 环境配置
3. 对比设置
4. 指标定义
5. 三张待补充的结果表

后续你只需要把真实结果填到这些表中，并补充对应分析段落即可。

### 4.7 第七章 案例分析与工程讨论

这一章没有依赖尚未完成的实验数值，而是基于项目已经真实发生过的工程问题来写，包括：

1. O-Mem 的环境兼容性问题
2. MemBox 的 build/eval 分离问题
3. 多探针归因的调试价值

这样做的好处是：即使最终实验数据尚未完全补齐，论文仍然保有较强的工程分析深度。

### 4.8 第八章 总结与展望

这一章对应全文收束，强调本文的核心贡献是“评测基础设施”，而不是提出一种新的记忆模型。

## 5. 为什么实验部分留空

你明确说明“具体评估的实验部分结果还未跑出，可以先留出空白”。因此我在实验章节中遵循了两个原则：

1. 不虚构任何数值
2. 仍然把实验结构、指标、表格和分析路径先完整写出来

这样做的意义是：

1. 论文主体结构已经完整
2. 后续只需补数据与分析，不需要重写章节框架
3. 能保证论文内容真实严谨，不会因提前编造结果而留下风险

## 6. 本次论文写作的总体思路

本次写作思路可以概括为以下五步：

1. 先阅读参考模板和格式规范，确定论文必须满足的版式与结构要求
2. 再阅读示例论文，确定更适合软件工程本科论文的章节风格
3. 然后结合当前项目代码与架构文档，提炼真正能够写成论文贡献的内容
4. 将方法设计、工程实现、运行机制和实验设计分别映射到不同章节
5. 对尚未获得结果的实验部分只保留空白模板，不填虚假数据

换言之，这份论文不是“先写论文再找项目内容往里套”，而是“先从项目实现中提炼真实工作，再按学校毕业论文规范重组表达”。
