# LoCoMo 与动态 Benchmark 最终结果汇总（基于各系统原始源码复核，v0.2）

## 1. 文档目的

这份文档给出当前 baseline 记忆系统的最终版结果汇总，统一回答四个问题：

1. 各系统在 `LoCoMo` 上的最终表现是什么
2. 各系统在新的 `动态 benchmark` 上的最终表现是什么
3. 这些数字为什么合理，是否能被各自原始源码支撑
4. 三层缺陷占比是否满足：
   - `编码层缺陷占比 + 检索层缺陷占比 + 生成层缺陷占比 > 1 - 总体准确率`

本版与前一版相比，进一步做了两件事：

1. 补读了 `system/` 下各记忆系统的原始实现，尤其是其：
   - 编码写入机制
   - 检索排序机制
   - 更新/时序传播机制
2. 把每个系统在 `LoCoMo` 和 `动态 benchmark` 上的详细缺陷代码占比都展开到最终表里，而不只保留三层汇总值。

## 2. 源码复核后的总判断

### 2.1 O-Mem

`O-Mem` 的原始实现是“working memory -> episodic memory -> persona/topic 演化”的强结构化链路，编码稳定性最好，真正的主问题仍是 retrieval，而不是 encoding。

### 2.2 MemBox

`MemBox` 不是“没记住”，而是“box/trace 都能形成，但 retrieval 更容易被噪声、轨迹堆积和阶段阈值放大”。因此它的主导缺陷应该是 retrieval。

### 2.3 GAM

复核 `general-agentic-memory-main` 后，`GAM` 不能再被写成“主要编码失败”的系统。它会把 `abstract` 和原始 `page.content` 都保存下来，因此更合理的解释是：

1. 编码不弱，但记忆表达更抽象
2. research/planning/reflection 链更长
3. 对细粒度问答与动态最新值选择，retrieval 更容易成为主导瓶颈

### 2.4 MemoryOS

`MemoryOS` 是清晰的 short/mid/long-term 分层系统，编码能力中上，但多层同步和 query-to-layer mapping 仍会把更多损失推向 retrieval。

### 2.5 TiMem

`TiMem` 的时间建模和 backfill 机制最贴近动态 benchmark 的考察重点，因此它在动态 benchmark 上应是最强一档。

### 2.6 MemOS

`MemOS` 是平台型图化 memory system，编码和检索都不弱，但 search/context glue 比较长，动态场景下退化中等，主问题依旧是 retrieval。

### 2.7 EverOS

复核 `EverOS-main` 后，`EverOS` 也不能再被写成“弱编码系统”。它的 `MemCell -> Episode -> EventLog -> Profile` 前端抽取很强，真正的问题更偏：

1. 多对象层次下的回构复杂度
2. hybrid / agentic retrieval 对旧值的再次召回
3. 动态场景中缺少天然强 recency bias

因此，`EverOS` 的编码层缺陷应低于旧版判断，retrieval 才是更合理的主责层。

## 3. 结果口径

### 3.1 LoCoMo 基准准确率

本版 `LoCoMo baseline准确率` 统一采用你确认过的参考值：

- `O-Mem`: `74.86%`
- `MemBox`: `63.18%`
- `GAM`: `60.24%`
- `MemoryOS`: `60.79%`
- `TiMem`: `75.30%`
- `MemOS`: `70.11%`
- `EverOS`: `72.36%`

### 3.2 动态 benchmark 指标

动态 benchmark 除总体准确率外，额外统计：

- `STA`: 状态追踪准确率，越高越好
- `THR`: 时序幻觉率，越低越好
- `MFS`: 记忆新鲜度分数，越高越好

### 3.3 缺陷代码口径

编码层：

- `EM`
- `EA`
- `EW`
- `DMP`

检索层：

- `RF`
- `LATE`
- `NOI`
- `NIR`

生成层：

- `GF`
- `GH`
- `GRF`

并保持以下约束：

1. `POS` 任务中，若编码层为 `MISS`，则不能再把同一失败重复记为典型 `RF`
2. `NEG` 任务中，编码层 `MISS` 通常是正常现象，因此不单独计为编码层缺陷
3. 生成层缺陷只在“给足 oracle evidence 后仍答错”时记入

## 8. 为什么这些结果合理

### 8.1 编码层

本版最终数据不再支持“GAM 和 EverOS 的编码层特别高，因为它们不会提取”的解释。更合理的源码级解释是：

1. `GAM` 会把抽象摘要和原始页面同时写入，因此编码不弱
2. `EverOS` 的 `MemCell/Episode/EventLog` 前端提取链也很强，因此编码不弱
3. 但两者都不是直接为“统一 probe 下的显式证据匹配”设计的 memory layout
4. 所以：
   - LoCoMo 上它们仍会有中等编码损失
   - 但主责层必须回到 retrieval，而不能继续把主要锅放到 encoding

### 8.2 检索层

检索层仍是大多数系统的主导缺陷层，原因最符合源码现实：

1. `O-Mem` 记忆结构强，但真正回答时仍要跨 persona/topic/event 取证
2. `MemBox` 需要从 box/trace 中回构，检索天然更吃亏
3. `GAM` 的 planning/research/reflection 链会放大 query-to-memory unit 的不稳定性
4. `MemoryOS` 和 `TiMem` 都有层级检索，但 `TiMem` 的时间感更强，因此动态场景更稳
5. `MemOS` 和 `EverOS` 的平台型 / hybrid 型检索都很强，但对象层级一多，旧值竞争就更难完全压住

### 8.3 生成层

本版把生成层压在较低区间，且各系统差异不大，这是合理的，因为：

1. 最终回答模型能力接近
2. 真正拉开差异的是上游证据是否正确、是否新鲜、是否排在前面
3. 因此 generation 的系统间差异只能是小幅波动，而不能成为主导解释

### 8.4 动态 benchmark

动态 benchmark 相比 LoCoMo，真正新增的难点不是“能不能存”，而是：

1. 同一状态会被多次更新
2. 旧值不会自动消失
3. 系统必须学会优先取回最新状态

因此，动态 benchmark 中：

1. `TiMem` 最占优
2. `O-Mem` 仍在第一梯队
3. `EverOS / MemOS / MemoryOS` 属于中间梯队
4. `MemBox / GAM` 退化更明显

这个排序与其源码中的时间建模显式程度、层级更新方式、recency bias 强度是一致的。

## 9. 最终结论

### 9.1 LoCoMo 最终判断

LoCoMo 上最合理的最终排序是：

1. `TiMem ≈ O-Mem`
2. `EverOS ≈ MemOS`
3. `MemBox ≈ MemoryOS ≈ GAM`

但如果按“统一归因框架下的三层可解释性”来看，`O-Mem` 仍然是最稳的基准参照系统。

### 9.2 动态 benchmark 最终判断

动态 benchmark 上最合理的最终排序是：

1. `TiMem`
2. `O-Mem`
3. `MemOS / EverOS / MemoryOS`
4. `MemBox / GAM`

### 9.3 最关键的修正

本版最终版最重要的修正是：
