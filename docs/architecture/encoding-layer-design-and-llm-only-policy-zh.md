# 编码层设计说明与需求对照（含「全程 LLM、禁止降级」策略）

**文档性质**：设计与方案说明；**不包含代码修改**。  
**读者假设**：已阅读 `docs/最终指标.md` 中编码层状态空间与缺陷定义。

---

## 1. 编码层要解决的问题（与你的表述对齐）

**目标**：判断**当前被测记忆系统**在内部状态中，是否**真实存在**回答当前问题所依赖的证据——即「标准答案所依据的信息，有没有以可观测形式进入该系统的记忆库」。

**评估时可用的信息**（优于纯黑盒）：

- 当前问题 **Q**；
- 与 Q 对齐的 **标准证据**（数据集中通常体现为 `f_key`、`evidence_texts`、`evidence_with_time`、`oracle_context` 等）；
- 任务类型 **POS / NEG**（拒答题上「不应存在相关记忆」是另一条判定分支）。

因此编码层本质上是一个 **「系统记忆内容 vs 证据契约」** 的符合性判定问题，而不是「模型最终答没答对」。

---

## 2. 「如何判断」——需要可操作定义

在实现前必须先固定 **Match 的语义**，否则不同系统无法对比、同一系统也不可复现。

### 2.1 建议拆成两个子问题

1. **观测获取（Observability）**  
   评估器能拿到什么视图来描述「系统里现在有什么」？  
   - 可能是：全量导出、分页导出、按 key 扫描、只读 API、调试快照等。  
   - **不同存储形态**（JSON 文件、向量库、图、自定义 heap）决定**导出形态**，这是适配器责任。

2. **符合性判定（Conformance）**  
   在给定观测下，证据是否「被认为已存储」？  
   - POS：是否存在与 **gold 证据语义等价** 的存储内容（允许表述差异、时间格式差异、轻微指代）。  
   - NEG：是否**不应**出现会诱导错误回答的「伪相关」记忆（与 `最终指标.md` 中 DIRTY / DMP 一致）。

### 2.2 为什么几乎必然需要「每系统适配」

底层结构不同会导致：

- **同一语义事实**在 A 系统是「一条 working memory 记录」，在 B 系统是「向量 chunk + metadata」；
- **遍历与候选生成**成本不同（全量导出不可行时，只能边界查询 + 采样）；
- **时间、说话人、会话 id** 等字段是否保留、如何编码各不相同。

因此：**「导出/候选构造」必须在适配器层按系统定制**；**「在给定候选与全库视图下如何判 EXIST/MISS/CORRUPT/DIRTY」可以尽量收敛到框架内统一流程（尤其用 LLM 统一裁判）**。

## 3. 编码层应如何设计（推荐架构）

下面给出一种与「通用框架 + 最薄适配器」兼顾的结构，并与你的两点诉求（混合检索进框架、减少适配器负担）对齐。

### 3.1 职责划分

| 组件 | 职责 |
|------|------|
| **适配器（系统相关）** | 1）`export_full_memory`：在评测允许的成本下导出**可序列化**的记忆视图；2）`find_memory_records`：在系统能力范围内做**与 Q / F_key 相关的候选检索**（可调用原生索引、scan、SQL、向量检索等）；3）可选：`hybrid_retrieve_candidates` 仅在适配器已实现「系统原生高召回检索」时提供，否则可不实现。 |
| **评估核心（系统无关）** | 1）定义**统一观测合同**（记忆条目的 `id/text/meta`）；2）提供**可选的通用混合候选接口**（例如：关键词 + 轻量语义代理，仅当适配器未提供足够召回时启用——注意：通用接口往往是近似，不能替代各系统原生召回）；3）编排 **LLM 编码裁判** 的输入（Q、证据、候选、任务类型）；4）**禁止在严格模式下用规则替代 LLM 结论**。 |

### 3.2 推荐判定流程（逻辑顺序）

对单个样本、在已有 `run_ctx` 上：

1. **M ← `export_full_memory(run_ctx)`**  
   - 得到全库视图（或评测约定的截断/分页视图，但必须在报告里声明）。

2. **C_enc ← 候选集合**  
   - 优先：`hybrid_retrieve_candidates`（若存在且成功）；  
   - 否则：`find_memory_records(run_ctx, Q, F_key, M)`；  
   - 框架侧**仅在非严格或显式允许时**才可追加通用候选（见第 6 节：你要求禁止降级后，这一项应关闭或仅用于调试管道）。

3. **LLM 编码裁判**  
   - 输入：`Q`、`task_type`、gold 证据（`evidence_with_time` 或 `f_key`）、`C_enc`（及必要时 `M` 的子集摘要）、**明确的 JSON schema**。  
   - 输出：`encoding_state ∈ {EXIST, MISS, CORRUPT_AMBIG, CORRUPT_WRONG, DIRTY}` + `defects` + `reasoning` + `matched_ids` 等。  
   - **失败策略**：解析失败 / schema 不满足 / 调用失败 → **整题 EVAL_ERROR**，不得静默改状态。

4. **与 `最终指标.md` 映射**  
   - 将 LLM 给出的状态与缺陷码 **EM/EA/EW/DMP** 对齐；若 LLM 标签与缺陷集合不一致，以**事先写死的映射表**为准并记录 `decision_trace`。

### 3.3 「混合检索放在编码层内部」是否更通用？

**可以，但要明确边界**：

- **框架内混合检索**（例如当前代码里的 token 重叠近似）优点是**适配器工作量小**；缺点是**召回与语义判定的可信度低于各系统原生能力**，容易与「真实是否写入向量库」脱节。  
- **更稳妥的通用化**是：框架定义 **「候选提供者」接口**，默认实现是弱混合，**生产评测强制使用适配器提供的原生高召回路径**；框架只负责 **LLM 统一裁判** 与 **证据打包协议**。

**结论**：你的方向「框架提供混合检索接口 + 适配器尽量薄」可行，但**薄适配器的前提是：该系统仍能通过少量 API 提供高召回候选**；否则编码层会系统性低估 EXIST。

## 4. 检索层与生成层（与你描述的对照）

以下为**目标语义**（与 `docs/最终指标.md` 及 `one-memory-system-full-evaluation-flow-v0.6.2` 一致）：

| 层级 | 应用输入 | 判定方式（目标） |
|------|----------|------------------|
| **检索** | `C_original`（**原生检索结果**）+ gold 证据 + Q | 以 **LLM 裁判为主**，结合 `rank/snr` 等**可计算属性**作为辅助证据写入报告。 |
| **生成** | `A_oracle`（完美上下文下系统生成）+ `A_online`（原生链路生成）+ `A_gold` | 以 **LLM 对比/判题**为主，输出 PASS/FAIL 及 GH/GF/GRF 等子类；**禁止**用简陋字符串相等替代语义判对（除非明确标注为「仅调试」且默认关闭）。 |

## 5. 现有实现是否能满足你的需求？（对照分析，不改代码）

### 5.1 已满足或部分满足的部分

- **适配器协议**已区分：`export_full_memory`、`find_memory_records`、可选 `hybrid_retrieve_candidates`，与第 3 节职责划分**方向一致**。  
- **编码层可接入 LLM**：`llm_judge_encoding_storage` 等已在 `eval_core/llm_assist.py` 中存在。  
- **严格配置项**在 `EvaluatorConfig` 中已存在：`require_llm_judgement`、`disable_rule_fallback`、`strict_adapter_call` 等，**具备**关闭规则兜底的开关语义。  
- **检索/生成**同样具备 LLM 裁判入口。

### 5.2 与「必须开启 LLM、真实代码不允许降级」的差距

以下差距来自**当前代码路径与脚本默认**，需在后续改造中消除（本文只列事实，不改代码）：

1. **冒烟脚本 `run_omem_stable_smoke_once.py` 显式设置 `use_llm_assist=False`**，且 `require_llm_judgement=False`、`disable_rule_fallback=False`，导致本次最小 demo 的归因**主要走规则路径**，与「检索/生成用 LLM 对比」的表述不一致。  

2. **`encoding.py` 中仍存在规则路径**：当 LLM 不可用或 strict 组合不当时，可能落到 `NEG` 规则分支、POS 的 `_fact_match` 路径等。要在产品上满足你要求，需要保证：**生产评测配置下这些路径不可达**（或仅保留在 `DEBUG`/`DEV` 模式）。  

3. **`llm_assist.py` 在 API 失败时返回 `None`**，上层再决定是否抛错；要严格「无降级」，应对失败统一上升为 **EVAL_ERROR**，且**不得**回退到规则判题。  

4. **编码候选与检索列表不一致**已在真实跑分结果中出现（编码 MISS 与检索 HIT 并存）：说明当前「通用混合/匹配」与「原生检索」**观测不一致**。这不一定违反指标文档，但**违背「尽可能判断系统是否存了证据」的直觉**；需要从设计上强制：**编码裁判输入必须包含与检索同源的观测或适配器声明二者关系**（方案项，见第 7 节）。

### 5.3 总体判断

- **架构分层与接口形态**：**基本能满足**你描述的「适配器 + 可选框架混合 + LLM 判定」方向。  
- **「全程 LLM、无降级」**：**当前默认路径尚未满足**；需要**配置策略 + 代码路径收敛**（后续迭代，不在本文实施）。  
- **「尽可能判断系统是否存了证据」**：**部分满足**；若继续允许规则匹配与弱混合为默认，会在复杂格式（日期、标点、指代）上**误判风险高**，与目标不符。

## 6. 「全程 LLM、禁止降级」解决方案（规范层，供后续实现）

### 6.1 定义「生产评测模式」Profile

建议新增或固定一种配置 profile（名称示例：`PROD_STRICT_LLM`），满足：

- `use_llm_assist = True`  
- `require_llm_judgement = True`  
- `disable_rule_fallback = True`  
- `strict_adapter_call = True`  
- `require_online_answer = True`  

**所有**对外报告、论文复现实验、CI 门禁，**仅承认**该 profile 下产出。

### 6.2 明确禁止的行为（红线）

1. LLM 调用失败、返回非 JSON、schema 校验失败 → **该样本记 `EVAL_ERROR`**，**禁止**用规则补判。  
2. 编码层 **禁止**在 `disable_rule_fallback=True` 时执行 `_fallback_find_records` 或等价全库扫描替代 LLM。  
3. 生成层 **禁止**用 `normalize` 字符串相等作为**最终** correctness（仅可作为 LLM 的 feature 或调试日志）。  
4. 脚本层 **禁止**将 `use_llm_assist=False` 作为默认 smoke / 正式实验入口；若保留，必须命名为 `*_rule_debug.py` 且文档声明**不用于结论**。

### 6.3 LLM 合同强化（与实现无关的设计要求）

- **统一 schema**：编码/检索/生成三类裁判输出均用 strict JSON + 必选字段。  
- **解析**：仅允许「去 fence + 提取平衡 JSON」类恢复；恢复失败算 **EVAL_ERROR**。  
- **证据回写**：每条判定必须带 `evidence_snippets` / `matched_ids`，满足 `最终指标.md` 对归因可解释性的方向（与 `REQUIREMENTS.md` R-002 的「引擎层证据归因」可一并规划）。

### 6.4 编码层观测一致性（解决 MISS vs HIT 张力）

可选强约束（推荐写入适配器契约）：

- **约束 A**：编码裁判的候选 `C_enc` **必须**包含「与 `retrieve_original` 同一次 native retrieval 导出的原始条目」（或适配器在 manifest 中声明「编码仅看全库导出，检索为另一视图」并说明合理性）。  
- **约束 B**：对 O-Mem 类系统，编码层候选默认 **合并** `export_full_memory` 与 **native retrieval top-N** 再送 LLM，以减少「存了但候选没扫到」的假 MISS。

## 7. 建议的后续实现顺序（仅路线图）

1. 统一 **PROD_STRICT_LLM** profile；修改 `run_omem_stable_smoke_once.py` 等入口默认值与之对齐。  
2. 收紧 `encoding.py` / `retrieval.py` / `generation.py`：在 strict 下 **删除或断言禁止**规则兜底路径。  
3. 收紧 `llm_assist.py`：失败不返回 `None` 作为「可继续」，改为抛错或结构化错误供 pipeline 记 `EVAL_ERROR`。  
4. 扩展 **adapter manifest**：声明编码候选来源、是否与检索同源。  
5. 补 **回归用例**：至少 1 个 POS + 1 个 NEG，强制 LLM on，期望输出含完整 `probe_results` 与错误样本的 `errors[]`。

## 8. 结论摘要

| 问题 | 结论 |
|------|------|
| 编码层应如何设计？ | **适配器负责可观测性与系统相关召回；框架负责统一证据合同与 LLM 裁判编排；可选框架弱混合仅作补充，不能替代原生高召回。** |
| 现有设计能否满足需求？ | **接口与分层方向满足；「全程 LLM、无降级」与「编码/检索观测一致」当前默认行为尚未满足，需按第 6～7 节收敛。** |
| 本文是否改代码？ | **否**；本文为后续改动的规范与对齐依据。 |
