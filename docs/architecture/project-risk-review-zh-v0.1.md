# 项目风险与漏洞复查说明（v0.1）

## 1. 复查范围

这次复查重点覆盖：

1. 新评估层 `src/memory_eval/eval_core`
2. 适配器层 `src/memory_eval/adapters`
3. `system/` 下三种记忆系统与评测接入相关风险

这里的“漏洞”主要按工程风险来理解，包括：

1. 逻辑漏洞
2. 状态污染
3. 配置陷阱
4. 可重复性风险
5. 评测公平性风险

## 2. 评估层已修复的关键风险

### 2.1 默认配置开箱即失败

旧问题：

1. 默认配置强依赖 LLM
2. 默认 API key 为空
3. 导致开箱即失败

当前处理：

1. 评估层默认改为本地可运行
2. LLM 辅助判定改为显式开启

### 2.2 RF 缺陷归因层次不清

旧问题：

1. 检索层会先打 `RF`
2. 再由归因层事后抑制

当前处理：

1. 检索层不再直接输出 `RF`
2. `AttributionAgent` 统一根据编码层状态补/抑 `RF`

### 2.3 NEG 生成判断过度依赖 gold 文本

旧问题：

1. NEG 下 `online_correct` 容易退化为字符串比对

当前处理：

1. NEG 下规则判断改为基于拒答行为
2. 使用拒答语义而不是 gold 文本包含关系

## 3. 当前仍需关注的项目风险

## 3.1 MemoryOS 原生生成有状态写回风险

风险：

1. 如果直接使用原生回答流程
2. 评测过程会反向写入记忆
3. 导致后续样本环境被污染

影响：

1. 破坏公平性
2. 破坏可重复性

建议：

1. 必须实现无副作用 `generate_online_answer`
2. 或 clone-state / rollback 机制

## 3.2 O-Mem fallback 与 real path 语义不一致

风险：

1. lightweight/fallback 并不等价于真实 O-Mem
2. 若混在正式评测中，会污染结论

建议：

1. 明确区分 native real adapter
2. debug fallback 只用于开发，不用于正式结果

## 3.3 MemoryOS 仍未正式接入

风险：

1. 当前项目宣称评估多个系统
2. 但 `system/MemoryOS` 尚未正式接入 adapter 层

影响：

1. 三系统对齐尚未完成

## 3.4 适配器层缺少统一能力声明

风险：

1. 评测层不知道一个 adapter 是“没有能力”还是“尚未实现”

建议：

1. 后续引入 capability 声明

## 3.5 重复适配器文件可能引发漂移

当前仓库中存在重复 O-Mem adapter 副本。

风险：

1. 维护时容易改一份忘一份
2. 导致行为漂移

建议：

1. 后续统一清理重复实现

## 4. 当前没有发现的高危问题

在本次复查范围内，没有在新评估层发现以下高危问题：

1. 动态执行外部输入代码
2. 直接使用 shell 执行不可信字符串
3. 评估层内显式反序列化不可信 pickle
4. 评估层内硬编码敏感密钥

但这不代表整个大仓库绝对没有其他问题，只表示：

> **在当前评估层与适配器相关路径上，没有发现明显高危代码执行类漏洞。**

## 5. 当前建议的安全/稳健性优先级

我建议按下面顺序继续处理风险：

1. 优先解决 MemoryOS 无副作用生成
2. 明确 O-Mem native / debug 语义分层
3. 补 MemoryOSAdapter
4. 引入适配器能力声明
5. 清理重复 adapter 文件

## 6. 结论

当前项目最大风险不在“传统安全漏洞”，而在：

1. 评测公平性
2. 状态污染
3. 配置陷阱
4. 系统适配不完全

其中评估层自身的关键配置与归因层次问题，这一轮已经修正；后续最值得优先关注的是适配器层和 `system/MemoryOS` 的接入风险。
